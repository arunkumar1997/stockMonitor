# DipSense — Debugging Guide

## Starting the App

### Backend
```bash
cd stockMonitor
bash start_backend.sh
# OR manually:
cd backend && source venv/bin/activate && uvicorn main:app --reload --port 8000
```

### Frontend
```bash
cd stockMonitor
bash start_frontend.sh
# OR manually:
cd frontend && npm run dev
```

Open: **http://localhost:5173**

---

## Common Issues

### ❌ "Cannot connect to backend" error in UI

**Cause**: Backend server isn't running or port 8000 is blocked.

**Fix**:
```bash
# Check if backend is running
lsof -i :8000
# Start backend
cd backend && source venv/bin/activate && uvicorn main:app --reload --port 8000
```

---

### ❌ Dashboard shows 0 stocks / all cards say "Data loading…"

**Cause**: The watchlist was not seeded into the DB, or the DB is empty.

**Fix**:
```bash
cd backend
source venv/bin/activate
python seed_db.py        # migrate stocks.json → SQLite watchlist
```

Then trigger a manual refresh for each stock via the `⟳` button, or use:
```bash
curl -X POST http://localhost:8000/api/scheduler/refresh/RELIANCE.NS
```

---

### ❌ Stock data is stale / not updating

**Cause**: Auto-refresh is disabled (to avoid yfinance rate limits). Data only refreshes manually.

**Fix**: Click the `⟳` button on the stock card's per-stock refresh (via `POST /api/scheduler/refresh/{symbol}`), or use the frontend's refresh button.

To check data freshness:
```bash
curl http://localhost:8000/api/scheduler/status | python3 -m json.tool
```
Look for `per_symbol_age_sec` — high values (>3600) mean stale data.

---

### ❌ yfinance rate limit / empty sparklines

**Cause**: Too many requests to Yahoo Finance in a short time.

**Fix**: Wait 5–10 minutes and retry. yfinance has a soft rate limit. The app is designed in **manual-refresh-only mode** to avoid this. Never trigger `refresh_all` in rapid succession.

Check the backend terminal for lines like:
```
[scheduler] ✗ SYMBOL error: ...
```

---

### ❌ Soft-deleted stock doesn't appear in Trash

**Cause**: The stock was hard-deleted (old behaviour before the DB migration) or the `deleted_at` column isn't set.

**Fix**: Check directly in the DB:
```bash
cd backend
sqlite3 stock_data.db "SELECT symbol, deleted_at FROM watchlist WHERE deleted_at IS NOT NULL"
```

---

### ❌ Theme doesn't persist across refresh

**Cause**: `localStorage` is cleared or the key is wrong.

**Check in browser DevTools**:
```js
localStorage.getItem("themeMode")  // should be "dark", "light", or "auto"
```

## Useful API Endpoints for Debugging

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/health` | GET | Server + scheduler status |
| `/api/stocks` | GET | Active watchlist from DB |
| `/api/stocks/deleted` | GET | Soft-deleted stocks |
| `/api/dashboard` | GET | Full analysis data for all active stocks |
| `/api/scheduler/status` | GET | Refresh state + per-stock data age |
| `/api/scheduler/refresh/{symbol}` | POST | Manually refresh one stock |
| `/api/stocks/{symbol}/restore` | POST | Restore a soft-deleted stock |
| `/api/stocks/{symbol}/purge` | DELETE | Hard-delete from all tables |

**Interactive API docs** (auto-generated by FastAPI):
```
http://localhost:8000/docs
```

## Inspecting the SQLite Database

```bash
cd backend
sqlite3 stock_data.db

-- Active watchlist
SELECT symbol, name, sector FROM watchlist WHERE deleted_at IS NULL;

-- Soft-deleted stocks
SELECT symbol, datetime(deleted_at, 'unixepoch', 'localtime') FROM watchlist WHERE deleted_at IS NOT NULL;

-- Cache freshness
SELECT symbol, datetime(updated_at, 'unixepoch', 'localtime') as last_updated FROM stock_cache ORDER BY updated_at DESC;

-- Full analysis JSON for one stock
SELECT result_json FROM stock_cache WHERE symbol='RELIANCE.NS';
```

## Reading Backend Logs

The backend prints to stdout. Key log lines:

| Prefix | Meaning |
|--------|---------|
| `[store] DB ready at ...` | Database initialized OK |
| `[scheduler] Started — manual-refresh-only mode` | Scheduler up, no auto-refresh |
| `[scheduler] Refreshing SYMBOL …` | Refresh started for a stock |
| `[scheduler] ✓ SYMBOL saved` | Refresh succeeded |
| `[scheduler] ✗ SYMBOL error: ...` | Refresh failed (check yfinance) |
| `[seed] Done!` | seed_db.py completed |

## Frontend DevTools

Open **Chrome DevTools → Console** and look for:
- **Network tab**: Check failed API calls (red rows) — see request/response bodies
- **Application → Local Storage → localhost:5173**: Check `themeMode`, `collapsedSectors`
- **Console**: React warnings (minor `h6 in h2` nesting warning from MUI Dialog is harmless)
